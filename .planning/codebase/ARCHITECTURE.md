> Auto-generated by gsd-codebase-mapper | Focus: arch | 2026-04-03

## Summary

Static portfolio site with zero build step, deployed directly to GitHub Pages from the repo root. The site uses two independent design systems: Obsidian (dark, main homepage) and Google-inspired (light, career-content sub-pages). Content is data-driven on the homepage via `content/data.json` and statically authored HTML in `career-content/`.

## Pattern Overview

**Overall:** Static site with file-based routing (no SPA, no router, no framework)

**Key Characteristics:**
- Every URL maps 1:1 to an HTML file on disk
- No server-side logic, no API calls, no database
- Two design systems coexist intentionally, with a bridge stylesheet for gradual migration
- Homepage fetches `content/data.json` at runtime to render dynamic sections (POCs, articles, videos)
- Career-content pages are fully standalone HTML documents with their own CSS/JS stack

## Layers

**Presentation Layer (Homepage):**
- Purpose: Single-page scrollable portfolio with hash-based navigation
- Location: `index.html`, `css/obsidian.css`, `js/site.js`
- Contains: Hero, metrics, POCs, portfolio cards, creative gallery, contact form
- Depends on: `content/data.json` for dynamic content, YouTube embeds for video
- Used by: Root URL visitors

**Presentation Layer (Career Content):**
- Purpose: Multi-page content hub with articles, case studies, portfolio items, and interactive tools
- Location: `career-content/` (all subdirectories)
- Contains: Standalone HTML pages, each with their own `<head>` and full document structure
- Depends on: Shared CSS at `career-content/css/`, shared JS at `career-content/js/`
- Used by: Links from homepage and direct navigation

**Data Layer:**
- Purpose: Structured content for homepage dynamic rendering
- Location: `content/data.json`
- Contains: Hero info, metrics, video list, case study metadata, article metadata, POC entries
- Depends on: Nothing
- Used by: `js/site.js` (fetched at runtime via `fetch()`)

**Asset Layer:**
- Purpose: Images, PDFs, and favicon
- Location: `images/`, `career-content/*/images/`, `content/publications/`, `favicon.svg`
- Contains: Profile photo, article hero images, placeholder SVGs, PDF publications

## Data Flow

**Homepage Load:**

1. Browser requests `index.html` from GitHub Pages CDN
2. `css/obsidian.css` loads (Geist fonts via Google Fonts CDN)
3. Static HTML renders immediately (hero, nav, structure)
4. `js/site.js` executes: fetches `content/data.json`, populates POCs section, articles section, and video carousel
5. User interacts via hash-based scroll navigation (`#hero`, `#pocs`, `#portfolio`, `#creative`, `#contact`)

**Career Content Page Load:**

1. Browser requests e.g. `career-content/articles/automation-strategy-article.html`
2. Inline `<script>` sets theme class on `<html>` to prevent FOUC (flash of unstyled content)
3. `career-content/js/loader.min.js` applies saved theme from `localStorage`
4. `career-content/css/style.min.css` loads (Roboto font via Google Fonts CDN)
5. `career-content/js/main.min.js` initializes theme toggle and mobile nav

**Resource Page Load (Self-Contained):**

1. Browser requests e.g. `career-content/resources/roi-calculator.html`
2. Page contains ALL styles and scripts inline — no external CSS/JS dependencies
3. Has its own color variables (a third palette variant, dark-themed)

**State Management:**
- Theme preference stored in `localStorage` under key `arc-theme` (homepage) and `theme` (career-content pages) -- **these are inconsistent**
- No other client-side state persistence
- No cookies, no session storage, no authentication

## Key Abstractions

- Files: `css/obsidian.css` (1674 lines)
- Pattern: CSS custom properties in `:root`, 4px spacing grid, three-tier text hierarchy, single cyan accent
- Fonts: Geist (sans), Geist Mono (mono)

- Files: `career-content/css/obsidian-bridge.css`

## Navigation Architecture

**Homepage:** Hash-based anchor navigation within a single page
- `#hero`, `#pocs`, `#portfolio`, `#creative`, `#contact`
- Smooth scroll handled by `js/site.js`

**Homepage → Career Content:** Standard `<a href>` links with `target="_blank" rel="noopener"`
- All career-content links open in new tabs
- No back-navigation from career content to homepage built into career-content pages

**Career Content pages:** Each is a standalone document
- Index pages use JavaScript variables (`PAGE_TITLE`, `REL_PATH`, `CANONICAL_PATH`) set per-page for meta generation
- No shared navigation component — each page defines its own header/nav

## Error Handling

**Strategy:** None (static site). 404s are handled by GitHub Pages default behavior.

**Patterns:**
- `js/site.js` uses defensive checks (`if (!carousel || !progressFill) return`) before DOM operations
- `localStorage` access wrapped in try/catch in theme scripts
- No custom 404.html page

## Cross-Cutting Concerns

**SEO:**
- OpenGraph and Twitter Card meta tags on `index.html`
- `sitemap.xml` enumerates all pages with priorities
- `robots.txt` present
- Career-content pages define `CANONICAL_PATH` for SEO

**Security:**
- CSP headers defined in `_headers` file and maintained by `.github/workflows/csp-header.yml`
- Allows: `unsafe-inline` for scripts/styles, YouTube frames, Google Fonts
- `object-src 'none'` blocks plugins

**Performance:**
- `rel="preconnect"` for Google Fonts on all pages
- `rel="preload"` for critical CSS/JS on career-content pages
- Lazy YouTube loading (thumbnail → iframe on click) in `js/site.js`
- No image optimization pipeline — images served as-is

**Accessibility:**
- Skip-link on homepage (`<a href="#main" class="skip-link">`)
- `aria-label` attributes on social links and theme toggle
- `lang="en"` on all HTML documents

**Pipeline:** GitHub Actions (`.github/workflows/deploy.yml`)
- Trigger: Push to `main` branch or manual `workflow_dispatch`
- Process: Checkout → Configure Pages → Upload entire repo root → Deploy
- No build step, no minification, no bundling
- Entire repo root is the artifact (includes `.planning/`, `.claude/`, etc.)
